**Результат проверки непротиворечивости FSM и достаточности заголовков**

Анализ конечного автомата подтверждает, что архитектура **полностью непротиворечива** и соответствует принципам stateless-модели. Заголовки писем переносят исчерпывающий контекст для каждого перехода FSM.
Обеспечивается достаточная изоляция (сообщения маршрутизируются в нужные треды и mailbox-ы):

1.  **Изоляция и возврат субагентов:** непрерывная цепочка `In-Reply-To` и маркеры `subagent_intent` / `subagent_end` гарантируют точный возврат из субагента в `ingress` родителя. Классификатор `depth` по IRT (§1 плана рефакторинга) в `egress_router` определяет, находится ли ответ внутри незакрытого субагента.
2.  **Восстановление HITL:** Построение базируется на сохранении состояния в union notmuch index'е (durable `stages/<stage>/Maildir/cur/<id>:2,S` после `nm_settle()`, [`INDEX.md §5.5.3`](INDEX.md#553-notmuch-consistency-через-notmuch2mutabletagset)). Возврат от пользователя (без внутренних заголовков) безошибочно восстанавливает контекст через `notmuch`-lookup прямого родителя по `In-Reply-To`: IRT-обход вверх → проверка `From: cli_hitl_out@localhost`.
3.  **Единая точка завершения:** `egress_router` — единственный узел, определяющий маршрут финального ответа. При `depth > 0` (незакрытый субагент по IRT) → `subagent_end` → восстановление hop/cap родителя → `ingress`. При `depth == 0` → внешний ответ пользователю. Создание новых фреймов (с изолированным hop/cap) — через `subagent_intent`.

## Термины

  - **Фрейм (Frame / Execution Frame)** — изолированный контекст исполнения (аналог стекового фрейма в программировании). Обладает собственным бюджетом (`X-Threlium-Hop-Budget`) и областью видимости (`X-Threlium-Capabilities`). Границы определяются маркерами `subagent_intent` (начало) и `subagent_end` (завершение) в IRT-цепочке.
  - **MainAgentLevel0 (L0)** — корневой агент пользовательского диалога (`main()`), исполняющийся в корневом фрейме.
  - **SubagentLevel1 (L1)** — субагент первого уровня вложенности (`sub1()`), исполняющийся во вложенном фрейме.
  - **SubagentLevel2 (L2)** — субагент второго уровня вложенности (`sub2()`). Рекурсия фреймов не ограничена количеством уровней.
  - **`U0`, `U1`, ...** — письма от/к пользователю (User).
  - **`L0_N`, `L1_N`, `L2_N`** — внутренние письма уровней 0, 1, 2. Индекс `N` — порядковый номер в рамках фрейма.
  - **`L0_HN`** — HITL-мост: письма, всплывающие из глубины стека наружу к пользователю через L0.
  - **`Route(Ux)` / `Route(Lx_y)`** — **значение** заголовка `X-Threlium-Route` на письме `Ux` или `Lx_y`: строка **b62(JSON)** с полем `channel` и channel-specific payload (см. [`MESSAGES.md` §8](MESSAGES.md#8-canonical-x-threlium-headers-glossary)). Это **не** то же самое, что `Message-ID` того же письма в угловых скобках.

## Логика роутеров

### `ingress_router` (входящий)

| # | Условие | Решение |
|---|---------|---------|
| 0 | Нет `In-Reply-To` | **Новый корневой диалог** (первое письмо от пользователя). Маршрутизируется в `enrich`. |
| 1 | Есть `In-Reply-To` → обход предков по IRT (1–N шагов) → найден **`From: cli_hitl_out@localhost`** | **HITL-возврат** (пользователь ответил на прерывание). Маршрутизируется в `cli_resume`. `In-Reply-To` исходящего письма = MID **входящего** сообщения (простой шаг, как для `enrich`), сохраняя линейную IRT-цепочку: `…→ archive(glue) → U1 → cli_resume`. `cli_resume` сам находит `cli_intent` по обходу IRT-предков (`find_cli_intent_maildir_path_from_in_reply_to_ancestors`). |
| 2 | Есть `In-Reply-To` → обход предков по IRT → не-HITL (reasoning/ingress/enrich/subagent_*) | **Обычное продолжение** (возврат из завершенного субагента, внутренний отказ/ошибка от `subagent_intent` или `cli_intent`, либо штатный ответ пользователя). Маршрутизируется в `enrich`. |

### `egress_router` (исходящий)

| # | Условие | Решение |
|---|---------|---------|
| 1 | `From: cli_hitl_out` | **HITL-мост наружу**. `resolve_route_for_egress_fsm_from_email` → `channel` из JSON wire → маршрутизация в `egress_<chan>@localhost`. Адрес пользователя (`<User-Address>`) роутер **не** извлекает — это делает сам `egress_<chan>`. |
| 2 | `From: reasoning` + **depth > 0** (IRT-классификатор: незакрытый `subagent_intent`) | **Внутренний возврат через `subagent_end`**. Маршрутизация в `subagent_end@localhost`: handler находит соответствующий `subagent_intent` по IRT, копирует hop/cap 1-в-1 с предка intent → `ingress`. |
| 3 | `From: reasoning` + **depth == 0** (корневой уровень) | **Внешний ответ (root)**. `resolve_route_for_egress_fsm_from_email` → `channel` из JSON wire → маршрутизация в `egress_<chan>@localhost`. Адрес пользователя извлекает `egress_<chan>`. |

**Приоритет правил:** порядок **1 → 2 → 3**. Сначала отделяется ветка **1** (`From: cli_hitl_out`, HITL-мост наружу); только если она не сработала, для `From: reasoning` вычисляется `depth` по IRT (классификатор: `subagent_end` → depth−1, `subagent_intent` → depth+1); `depth > 0` → **2** (subagent_end), `depth == 0` → **3** (внешний канал).

**Источник канала.** Определение канала и route-предка — по **самому новому** сообщению с ``tag:route`` и непустым ``X-Threlium-Route`` в notmuch-треде текущего письма (``NEWEST_FIRST``). В простом сценарии (одно пользовательское сообщение ``U0``) результат совпадает с IRT-обходом; при HITL-возврате (``U1`` — ответ на archive-glue, не находится в IRT-цепочке FSM) thread-based резолв находит ``U1`` — самый свежий ``tag:route``, обеспечивая корректный ``In-Reply-To`` во внешнем ответе.

**IRT egress_router для HITL vs external reply.**
- **HITL** (правило 1, `From: cli_hitl_out`): ``In-Reply-To`` исходящего письма = ``Message-ID`` **входящего** сообщения (cli_hitl_out output). Внутренняя IRT-цепочка **сохраняется**: `egress_<chan>` task → IRT → `egress_router` task → IRT → `cli_hitl_out` output. Полная линейная цепочка от `cli_resume` до `cli_hitl_out` (через archive-glue и U1): `cli_resume` →(IRT)→ `U1` →(IRT)→ `archive(glue)` →(IRT)→ `egress_<chan> task` →(IRT)→ `egress_router` →(IRT)→ `cli_hitl_out`. `classify_hitl_parent_notmuch` в `ingress_router` обходит эту цепочку и находит `From: cli_hitl_out@localhost`; `cli_resume` затем находит `cli_intent` по `find_cli_intent_maildir_path_from_in_reply_to_ancestors` по той же линейной цепочке.
- **External reply** (правило 3, `From: reasoning`, depth == 0): ``In-Reply-To`` заменяется на ``message_id_inner`` route-resolved предка — **самое новое** ``tag:route`` в notmuch-треде (при HITL-возврате — ``U1``; в простом сценарии — ``U0``). `egress_<chan>` использует этот IRT для формирования RFC-корректного ответа на последнее пользовательское письмо.

### Archive-связка (glue MID)

Каждая `egress_<chan>` стадия помимо отправки внешнего сообщения формирует **archive-запись** (`archive@localhost`). Ключевая роль archive — **связь внешнего идентификатора отправленного сообщения с внутренней IRT-цепочкой**:

1. `egress_<chan>` отправляет сообщение наружу и получает/генерирует **внешний идентификатор**:
   - **Telegram**: `message_id` из ответа Bot API.
   - **Matrix**: `event_id` из `RoomSendResponse`.
   - **Email**: свежий SMTP `Message-ID` (генерируется `make_msgid()` при отправке, отличный от FSM MID задачи `egress_email`).

2. Этот внешний идентификатор каноникализируется через `RfcMessageIdWire.from_native(...)` и становится `glue_message_id_wire` — **`Message-ID` archive-записи** (`build_egress_sent_record_to_archive`).

3. Когда пользователь отвечает на отправленное сообщение, мост каноникализирует `In-Reply-To` ответа тем же `from_native(...)`, получая **тот же canonical MID**, что у archive-записи. Тем самым `U1.In-Reply-To` → archive-запись (glue).

4. `In-Reply-To` самой archive-записи указывает на `Message-ID` задачи `egress_<chan>`, которая в свою очередь (при HITL) сохраняет IRT-цепочку до `cli_hitl_out`:

   ```
   U1  →(IRT)→  archive(glue)  →(IRT)→  egress_<chan> task  →(IRT)→  egress_router  →(IRT)→  cli_hitl_out
   ```

5. `ingress_router` при получении `U1` вызывает `classify_hitl_parent_notmuch`, который обходит эту цепочку предков и находит `From: cli_hitl_out@localhost` → маршрутизация в `cli_resume`.

**Единый пайплайн для всех каналов:** `egress_<chan>` → отправка внешнего сообщения → получение/генерация внешнего идентификатора → `RfcMessageIdWire.from_native(NativeId(...))` → `glue_message_id_wire` → `build_egress_sent_record_to_archive(...)` → `archive@localhost`. Разница только в источнике внешнего идентификатора: Telegram/Matrix получают его от сервера (`message_id`, `event_id`), email — генерирует свежий `make_msgid()` при отправке. Для всех трёх каналов glue MID **уникален** и отличен от FSM MID задачи `egress_<chan>` — archive-запись является единственной точкой входа ответа пользователя в notmuch-тред.

### Состояния памяти (вне субагентного стека)

Помимо строгих маршрутизаторов и цепочки делегирования, описанных в матрице ниже, FSM содержит выделенные состояния для фиксации знаний:
- `global_memory@localhost`: Формирование абстрагированных, фундаментальных знаний пользователя.
- `thread_memory@localhost`: Сохранение промежуточных выжимок контекста текущего диалога.

Это **полноценные переходы** графа (см. [`MEMORY_TABLE.md`](MEMORY_TABLE.md)), инициируемые LLM (`reasoning`); они **не** участвуют в логике субагентов — только `subagent_intent` (маркер начала, изоляция hop/cap), `subagent_end` (маркер завершения, восстановление hop/cap) и `egress_router` (depth-классификатор) работают с границами субагентов. Индексация LightRAG (`schedule_index_pending` на выделенном asyncio-loop в `threlium-engine`, single writer на `lightrag/working_dir/`, settled-сообщения после `nm_settle()`) описана в [`MESSAGES.md` §5](MESSAGES.md#5-stage-worker-и-lightrag-worker) и [`INDEX.md` §5b / §7](INDEX.md#5b-lightrag-worker). Контекст и интеграция в общей архитектуре — [`ARCHITECTURE.md §4.2`](ARCHITECTURE.md#42-состояния-памяти-краткая-сводка) (сводка) и [`MEMORY_TABLE.md`](MEMORY_TABLE.md) (пошаговые матрицы).
