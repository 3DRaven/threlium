# Формирование плана реализации — исследование и план

Skill для агента: **перед любой нетривиальной правкой** (новая стадия FSM, контракт enrich/context, bridge, типы, e2e-сценарий, рефакторинг сквозного потока) сначала провести **четыре обязательных исследования через подагентов**, затем выдать **план реализации** поставленной задачи. Код, стабы и коммиты — **только после** утверждения плана пользователем или явного «делай по плану».

Нормативная база: карта документов — [`docs/ARCHITECTURE.md`](../ARCHITECTURE.md) (таблица источников истины), типы — [`docs/TYPES.md`](../TYPES.md), соседние контракты — [`docs/FSM.md`](../FSM.md), [`docs/CONTEXT_CONTRACT.md`](../CONTEXT_CONTRACT.md), [`docs/MESSAGES.md`](../MESSAGES.md). Правило «не дублировать рядом стоящие компоненты» — `.cursor/rules/context.mdc`.

---

## 1. Границы ответственности

| Зона | Политика |
|------|----------|
| **Эта skill** | Исследование + **текстовый план** (шаги, файлы, риски, проверки). Без правок в git до завершения фазы 5, если пользователь не сказал иначе. |
| **Тривиальные правки** (опечатка в комментарии, однострочный фикс по уже согласованному плану) | Skill можно сжать до одного подагента «код» + краткий план из 3 пунктов; **четыре фазы** всё равно желательны при касании FSM/types/bridge. |
| **Продуктовый код** | В плане перечислять **конкретные** пути и функции; в реализации — минимальный diff (см. user rules). |
| **Документация** | План указывает, **какие** строки/разделы в `/docs` обновить после кода; не переписывать документы целиком (`.cursor/rules/docs.mdc`). |
| **E2e** | В плане — какой тест/стаб затронут; отладка по [`mailflow-e2e-wiremock-sut.md`](mailflow-e2e-wiremock-sut.md), если задача касается mailflow. |

**Не отвлекать пользователя** вопросами «начинать ли исследование», пока не завершены фазы 1–4 или нет жёсткого блокера (нет доступа к репо, задача не сформулирована).

## 2. Жёсткие ограничения

### 2.1. Порядок работы

- **Запрещено** писать код или менять стабы **до** завершения всех четырёх исследований (§3) и черновика плана (§4).
- **Запрещено** объединять фазы в один «общий» подагент без отдельных отчётов по осям: код, типы, docs, актуальная логика.
- **Запрещено** вводить новый тип/VO, не сверившись с [`docs/TYPES.md`](../TYPES.md) и `threlium.types.__all__`.
- **Запрещено** планировать второй bridge/matrix/email путь, если есть общий контракт в `threlium.bridges` / `bridge.py` (см. context rule).

### 2.2. Подагенты (Cursor Task)

| Тип | Когда |
|-----|--------|
| `explore` | Поиск по репо, grep-подобные обходы, карта вызовов, «где уже сделано». |
| `generalPurpose` | Сводка нескольких файлов, сопоставление doc ↔ код, черновик таблиц. |

- Для **независимых** осей (§3.1–3.4) — **параллельно** до четырёх подагентов в одном сообщении.
- В промпте подагенту: **конкретная задача**, список путей-якорей, формат ответа (см. §3), запрет на правки файлов (`readonly: true` при только чтении).
- Родительский агент **синтезирует** отчёты; не копировать сырые простыни в чат пользователю — только структурированные выводы.

### 2.3. Unit-тесты

- В проекте **нет** unit-слоя; в плане проверки — **e2e** или ручные шаги из [`docs/TESTING.md`](../TESTING.md).

## 3. Обязательные исследования (до плана)

Каждая фаза заканчивается **кратким отчётом в чат** (шаблон §3.5). Только когда все четыре отчёта готовы — переход к §4.

### 3.1. Переиспользуемый код

**Цель:** не писать вторую реализацию того, что уже есть в `ansible/roles/threlium/files/scripts/threlium/`.

**Подагент `explore` (промпт-скелет):**

```
Задача: найти переиспользуемый код для: <формулировка задачи пользователя>.
Искать: threlium/ (states, bridges, common, runners, mime_reform, enrich, lightrag, types).
Для каждого кандидата: путь, функция/класс, 1–2 предложения контракта, кто вызывает (grep/usage).
Отдельно: есть ли симметричные каналы (email/matrix/telegram) и общий bridge.
Выход: таблица «переиспользовать | адаптировать | нет аналога».
readonly, без правок.
```

**Минимум проверить:**

- `threlium.bridges`, `threlium.common` (билдеры MIME), `threlium.mail`, `threlium.states.<соседние_стадии>`.
- Если задача про enrich/context — `build_context_backpack_multipart`, `graph_answer_view`, промпты `prompts/<stage>/`.
- Если задача про настройки — только `ThreliumSettings` / `load_settings()`, не новые env-VO.

### 3.2. Типы и [`docs/TYPES.md`](../TYPES.md)

**Цель:** новые поля/заголовки — через существующие VO; соблюдение границы msgspec и фабрик `_OptionalStrip*`.

**Подагент `explore` или `generalPurpose`:**

```
Задача: для <задача> сверить с docs/TYPES.md и пакетом threlium/types/.
Прочитать TYPES.md (принципы + релевантные §).
Поиск по репо: похожие *Wire, *B62, Notmuch*, FsmStage, ingress_hitl, EngineWire*.
Выход:
- какие существующие типы использовать (с именами классов);
- что запрещено (второй RfcHeader, NonEmptyStr на границе смысла, голые THRELIUM_*);
- если нужен новый тип — обоснование, что дубликата нет, база _core, модуль.
readonly.
```

**Обязательные инварианты из TYPES.md в плане:**

- strip/normalize на границе, не в `__post_init__` бизнес-модели;
- `FsmStage` ↔ `vars/main.yml` ↔ `STAGE_MAIN_MODULES` при новой стадии;
- `IngressRoute` / b62 только через VO-методы;
- present-or-None vs `parse` — явно в плане для optional заголовков.

### 3.3. Актуальная документация `/docs`

**Цель:** план не противоречит каноническим документам; знать, что обновить после реализации.

**Подагент `generalPurpose`:**

```
Задача: для <задача> прочитать релевантные docs/*.md.
Старт: ARCHITECTURE.md (таблица источников истины).
Добавить по смыслу: FSM, INDEX, MESSAGES, CONTEXT_CONTRACT, SUBAGENT_TABLE,
ORCHESTRATION, HOWTO_ADD_FSM_STAGE, TESTING, E2E_ISOLATION, TYPES.
Выход:
- список документов + §/якоря, релевантные задаче;
- цитаты ограничений (инварианты, запреты), которые план обязан соблюдать;
- расхождения doc vs код (если найдены) — пометить «сначала уточнить у пользователя».
readonly.
```

**Не дублировать** в плане длинные выдержки — ссылки на файл и § достаточно.

### 3.4. Актуальная логика кодовой базы

**Цель:** понять **текущий** поток выполнения (не устаревший briefing), точки входа и побочные эффекты.

**Подагент `explore`:**

```
Задача: проследить актуальный поток для <задача>.
Точки входа: runners/engine, engine_submit, states/<stage>.main, run_fdm, nm_settle,
threlium-dispatch, fdm pipe (если доставка).
Выход: numbered flow (5–15 шагов), список файлов по порядку, side effects
(notmuch теги, systemd units, HTTP к wiremock/litellm/lightrag).
Отметить терминальные стадии (return None, без run_fdm).
readonly.
```

**Согласовать** с отчётом §3.1: если «переиспользовать» и «текущий поток» расходятся — зафиксировать как риск в плане.

### 3.5. Шаблон отчёта по фазе (в чат)

```markdown
### Исследование: <код|типы|docs|логика>
- **Вывод в одном предложении:** …
- **Ключевые артефакты:** `path` (функция/§)
- **Решение для плана:** переиспользовать | расширить | новое (с оговорками)
- **Риски / открытые вопросы:** …
```

## 4. Формирование плана (только после §3)

Синтез четырёх отчётов → **один документ плана** в чат (и при необходимости `docs/briefing/<topic>_briefing.md` — только если пользователь просит файл).

### 4.1. Структура плана

| Раздел | Содержание |
|--------|------------|
| **Цель** | Что должно измениться для пользователя/FSM (1–3 предложения). |
| **Вне scope** | Что сознательно не трогаем. |
| **Переиспользование** | Таблица из §3.1 + §3.2 (код + типы). |
| **Контракты docs** | Список инвариантов из §3.3 (ссылки). |
| **Текущий vs целевой поток** | До/после по §3.4 (кратко, можно mermaid). |
| **Шаги реализации** | Нумерованный список: файл → действие → зависимость от предыдущего шага. |
| **Ansible / prompts** | Если нужны `vars/main.yml`, j2, wiremock_stubs. |
| **Проверка** | Конкретные e2e `nodeid` или ручные команды; без новых unit-тестов. |
| **Документация после кода** | Точечные правки в `/docs` (файл + §). |
| **Риски и откат** | FSM-зацикливание, notmuch, изоляция e2e. |

### 4.2. Критерии готовности плана

План можно считать готовым, если:

- [ ] Есть явное «переиспользовать X» или обоснованное «новый модуль Y».
- [ ] Нет нового типа без проверки TYPES.md.
- [ ] Указаны затронутые стадии FSM и терминальность `egress_*`.
- [ ] Указан способ проверки (e2e / journald / notmuch).
- [ ] Пользователь мог сказать «ок» или «правь шаг N» без чтения всего репо.

**После «ок»** — реализация по шагам; при отклонении от плана — остановиться и обновить план, а не тихо расширять scope.

## 5. Порядок работы (workflow)

Соблюдать **строго по шагам**.

| Шаг | Действие |
|-----|----------|
| **0** | Уточнить формулировку задачи у себя (из сообщения пользователя); зафиксировать **одну** цель. |
| **1** | Запустить **параллельно** подагентов §3.1–3.4 (четыре исследования). |
| **2** | Опубликовать четыре отчёта по шаблону §3.5. |
| **3** | Свести противоречия (код vs docs vs типы); при блокере — **один** вопрос пользователю с вариантами. |
| **4** | Выдать план §4.1; **не писать код**. |
| **5** | Дождаться явного согласования или прямой команды «реализуй» / «делай». |
| **6** | Реализация минимальным diff; e2e по [`mailflow-e2e-wiremock-sut.md`](mailflow-e2e-wiremock-sut.md) при необходимости. |
| **7** | Точечно обновить `/docs` из раздела плана. |

## 6. Когда skill обязательна

| Ситуация | Skill |
|----------|-------|
| Новая/изменённая FSM-стадия, bridge, enrich-контракт | **Да**, все 4 фазы |
| Новый VO / заголовок `X-Threlium-*` | **Да**, усилить §3.2 |
| Правка только e2e-стаба под уже согласованное поведение | Можно сократить до §3.4 + §3.3 (TESTING/E2E_ISOLATION) |
| Однострочный typo | Нет, если не затрагивает контракты |

## 7. Антипаттерны

- Начать с `states/enrich.py` до чтения CONTEXT_CONTRACT и `build_context_backpack_multipart`.
- Добавить `class FooWire` без поиска по `threlium/types/` и TYPES.md.
- План «сделаем рефакторинг всего bridge» без таблицы переиспользования.
- Один подагент «изучи всё» вместо четырёх осей — теряются типы и docs.
- Писать код «заодно пока исследую».
- Переписать целиком ARCHITECTURE.md или FSM.md после задачи.
- Игнорировать симметрию email/matrix/telegram при изменении одного канала.
